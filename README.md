# Toss Meeting Prototype

회의 시간을 고르는 일은 보통 “빈 시간 찾기” 문제처럼 보입니다.  
그런데 여러 사람의 캘린더를 실제로 맞춰보면, 더 어려운 순간은 빈 시간이 없을 때 찾아옵니다.

“그럼 누구 일정을 옮겨야 하지?”  
“이 부탁을 보내도 되는 상황인가?”  
“상대가 거절하면 다시 처음부터 계산해야 하나?”

이 프로토타입은 그 지점에서 출발했습니다.  
단순히 가능한 시간을 추천하는 것이 아니라, **회의가 막혔을 때 어떤 조정이 가장 덜 부담스러운지 계산하고, 그 요청을 안전하게 주고받는 경험**을 실험합니다.

## Live Demo

- Prototype: https://toss-meeting-prototype.vercel.app
- GitHub repo: https://github.com/juun9690-blip/toss-proto

## What I Designed

이 프로젝트의 핵심 질문은 하나였습니다.

> “일정을 자동으로 잡아주는 도구가 아니라, 조율의 부담을 줄여주는 도구라면 어떤 모습이어야 할까?”

기존 캘린더 도구는 시간이 겹치면 후보를 보여주거나, 전원 가능한 시간이 없으면 사실상 멈춥니다. 그 다음의 일은 사람이 떠안습니다. 누가 양보할 수 있는지 다시 보고, 회의 중요도를 설명하고, 상대가 부담스럽지 않게 메시지를 쓰고, 거절되면 다시 계산합니다.

저는 이 부담을 **재계산 비용**이라고 봤습니다.  
그리고 재계산 비용은 혼자 존재하지 않았습니다. 그 안에는 “남의 일정을 건드려도 되는가”라는 **관계 비용**도 같이 들어 있었습니다.

그래서 이 프로토타입은 세 가지를 실험합니다.

1. 전원 가능한 시간이 없을 때, 시스템이 최소 조정안을 계산한다.
2. 조정은 강제가 아니라 요청으로 전달한다.
3. 요청받는 사람도 거절하거나 다른 이동 시간을 고를 수 있어야 한다.

## Design Story

처음에는 문제를 단순하게 생각했습니다.  
“6명의 캘린더에서 모두 되는 시간을 잘 찾으면 되지 않을까?”

하지만 인터뷰와 리서치를 하면서 생각이 바뀌었습니다. 실제 회의 조율에서 사람들을 지치게 하는 건 검색 자체보다 검색이 실패한 뒤였습니다. 모두 되는 시간이 없을 때, 사람들은 머릿속으로 이런 계산을 다시 합니다.

- 필참자는 누구인가?
- 선택 참석자는 빠져도 되는가?
- 옮길 수 있는 일정은 무엇인가?
- 이 요청은 상대에게 부담스럽지 않은가?
- 거절당하면 다음 대안은 있는가?

이 과정은 캘린더 안에서 일어나지만, 사실은 사람 사이의 협상에 가깝습니다. 그래서 제품의 방향도 “자동 확정”이 아니라 “조율을 돕는 계산기”에 가까워졌습니다.

프로토타입의 골든 시나리오는 일부러 전원 가능한 시간이 없도록 설계했습니다. 사용자가 막히는 장면을 피하지 않고, 바로 그 막힌 지점에서 제품의 가치가 드러나게 하기 위해서입니다.

앱은 가능한 시간을 나열하는 대신 이렇게 말합니다.

> “이 시간은 모두 어렵지만, 김선우 님의 이동 가능한 일정을 한 칸 옮기면 전원이 참석할 수 있어요.”

여기서 중요한 건 “옮기면 됩니다”가 아닙니다.  
중요한 건 **왜 이 요청이 가장 작은 부탁인지**, 그리고 **상대가 이 부탁을 받아들일지 말지 선택할 수 있는지**입니다.

## What Changed Through Testing

초기 가설은 “요청받는 사람의 부담”에 집중되어 있었습니다. 실제로 AI UT와 대면 인터뷰에서 그 부담은 분명히 보였습니다. 특히 낮은 직급의 사람일수록 거절 자체보다 “어떻게 정중하게 거절하지?”라는 감정노동이 컸습니다.

그래서 응답 화면은 단순한 `수락 / 거절`에서 바뀌었습니다.

- 거절은 실패가 아니라 다음 조정안으로 이어지게 했습니다.
- “거절”이라는 날카로운 표현보다 “다른 시간이 좋아요”처럼 말할 수 있게 했습니다.
- 일정을 옮기는 경우에도 자동으로 밀리는 것이 아니라, 본인이 이동할 시간을 직접 고르게 했습니다.

그런데 실험을 더 하면서 반대편 부담도 보였습니다. 요청자 역시 망설였습니다.

“아직 회의가 확정된 것도 아닌데, 이 요청을 보내도 되나?”

이 발견이 중요했습니다. 관계 비용은 요청받는 사람만 지는 것이 아니었습니다. 요청자도 누군가의 일정을 건드리는 사회적 행동을 시작해야 했고, 그 행동의 정당성을 화면에서 확인하고 싶어했습니다.

그래서 제품은 점점 “빠르게 잡아주는 도구”가 아니라, **부탁의 비용을 계산하고, 그 부탁이 안전하게 오가도록 돕는 도구**로 정리되었습니다.

## Core Interaction

프로토타입의 핵심 흐름은 다음과 같습니다.

1. 회의 정보를 입력한다.
2. 참석자를 필참/선택으로 나눈다.
3. 전원 가능한 시간이 없다는 것을 확인한다.
4. 시스템이 최소 조정안을 제안한다.
5. 요청자는 근거를 보고 요청을 보낸다.
6. 수신자는 자신의 화면에서 요청의 영향과 대안을 확인한다.
7. 수락하면 회의와 이동 일정이 함께 확정되고, 거절하면 다음 조정안으로 돌아간다.

이 흐름에서 제품이 대신하는 것은 결정이 아닙니다.  
제품은 계산을 대신합니다. 결정은 여전히 사람이 합니다.

## Design Principles

### 1. 계산은 시스템이, 판단은 사람이

시스템은 어떤 시간이 가장 적은 조정으로 가능한지 계산합니다. 하지만 누군가의 일정을 강제로 바꾸지는 않습니다. 이 선을 지키는 것이 중요했습니다.

### 2. “가능한 시간”보다 “덜 부담스러운 부탁”

후보 시간의 우선순위는 단순히 빠른 시간이 아닙니다. 조정 없는 시간, 한 명만 조정하면 되는 시간, 관계 부담이 적은 요청 순서로 정렬됩니다.

### 3. 수신자에게 통제감을 남기기

요청받는 사람은 단순히 수락하거나 거절하는 사람이 아닙니다. 본인 일정의 이동 위치를 고를 수 있어야 하고, 어려우면 다른 시간을 제안할 수 있어야 합니다.

### 4. 민감한 계산은 조용히 보여주기

“누가 한가한가” 같은 신호는 잘못 보이면 감시처럼 느껴질 수 있습니다. 그래서 요청자 화면에서는 사람을 평가하는 언어를 피하고, 순서와 근거만 조심스럽게 드러내는 방향을 택했습니다.

## What I Learned

이 프로젝트에서 가장 크게 배운 것은, 일정 조율이 단순한 최적화 문제가 아니라는 점입니다.

캘린더의 빈칸은 수학적으로 찾을 수 있지만, 그 빈칸을 만들기 위해 누군가에게 부탁하는 순간 제품은 사회적 공간에 들어갑니다. 좋은 디자인은 그 사회적 비용을 없애겠다고 말하기보다, 어디서 발생하는지 보고, 사용자가 덜 외롭게 감당하도록 돕는 것에 가까웠습니다.

그래서 이 프로토타입은 완성된 서비스라기보다 하나의 실험 기록입니다.

- 문제를 “빈 시간 찾기”에서 “재계산 비용”으로 다시 정의한 실험
- 자동화가 사람의 결정을 침범하지 않도록 경계를 찾은 실험
- 요청자와 수신자의 비대칭 부담을 화면 흐름으로 다뤄본 실험
- 인터뷰와 UT에서 나온 발견을 바로 프로토타입에 반영한 실험

제가 보여주고 싶은 것은 정답 하나가 아니라, **문제를 좁히고, 반례를 만나고, 다시 설계를 고쳐가는 과정**입니다.

## Prototype Path

이 저장소는 React + Vite + TypeScript로 만든 웹 프로토타입입니다.

```bash
npm install
npm run dev
```

로컬 실행 후 터미널에 뜨는 주소를 브라우저에서 열면 됩니다. 보통 `http://localhost:5173` 입니다.

## Project Structure

- `src/data/mock.ts` — 참석자, 일정, 회의실 목업 데이터
- `src/logic/scheduling.ts` — 후보 시간 계산과 조정안 생성 로직
- `src/screens/` — 단계별 화면
- `src/components/` — 캘린더, 패널, 상태 표시 UI

## Current Scope

현재 프로토타입은 핵심 플로우 검증에 집중합니다.

- 실제 캘린더 연동 없음
- 로그인 없음
- 알림 발송 없음
- 모바일 최적화는 제한적
- 데이터는 골든 시나리오 검증을 위한 목업

그 대신 사용자가 “모두 되는 시간이 없을 때 제품이 무엇을 해주는지”를 끝까지 클릭해볼 수 있도록 만들었습니다.
